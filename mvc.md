## MVC 패턴
MVC 패턴은 디자인 패턴 중 하나이다.
### 디자인 패턴
디자인 패턴이란, 프로그램이나 어떤 특정한 것을 개발하는 도중에 생겼던 문제점들을 정리, 상황에 따라 간편하게 적용해서 쓸 수 있는 것을 정리해서 특정한 규약을 통해 쉽게 쓸 수 있는 형태로 만든 것을 말함.
예시로 어떤 앱을 만들어본다고 가정해보자. 이 앱을 유지보수를 하고, 다른 사람들과 공유를 하면서 만들어야 할 때, 먼저 우리는 깔끔하게 만들 수 있는 방법을 고안해야 됨. 이런 방법들을 명확하게 하지 않는다면? => 클래스 함수를 일일히 다 만드는 하드코딩 아닌 하드코딩 완성.
만약 jQuery라는 것을 사용하면, $('#lucid')로 DOM을 선택할 수 있지만, 순수 JS를 사용하게 되면, document.genElementsByid('lucid')라는 길고 복잡하게 써가면서 찾아야 됨.
만약에 어떤 데이터를 만들고, 데이터를 수정할 로직을 짤 때, 이 데이터를 보여주는 부분을 만들 때, 로직들이 분리가 되어 있지 않고, 한꺼번에 정의가 되어 있다면, 유지보수를 하기 어려움.
이것을 막기 위해 디자인 패턴이 나오게 되었고, '좀 더 쉽고 편리하게 사용' 을 위해 만들어진 특정한 방법들을 디자인 패턴이라고 함.
(디자인 패턴에는 스트래티지 패턴, 옵저버 패턴 등 여러가지가 있다.)

## MVC
MVC는 Model, View, Controller의 약자로, 하나의 앱, 프로젝트를 구성할 때, 그 구성 요소를 세 가지의 역할로 구분한 패턴
사용자가 컨트롤러를 조작하면, 컨트롤러는 모델을 통해 데이터를 가져오고, 그 정보를 바탕으로 뷰를 제어, 사용자에게 전달하게 됨.

### Model
애플리케이션의 정보, 데이터를 나타냄. 데이터베이스, 상수, 초기화값, 변수 등과 이러한 데이터들의 가공을 책임지는 컴포넌트(소프트웨어에서 재사용 가능한 독립적인 모듈)
규칙:
1. 사용자가 편집하기 원하는 모든 데이터를 가지고 있어야 함.
2. 뷰나 컨트롤러에 대해서 어떠한 정보도 알아서는 안 됨. (모델에서 화면 UI를 직접 조정해서 수정할 수 있게 뷰를 참조하는 내부 속성값을 가지면 안 됨)
3. 변경이 일어날 때, 변경 통지에 대한 처리 방법이 구현되어 있어야 함.

### View
input 텍스트, 체크박스 항목 등과 같이 사용자 인터페이스 요소를 나타냄. (stdio)
규칙:
1. 모델이 가지고 있는 정보를 따로 저장해서는 안 됨.
2. 모델이나 컨트롤러 등 다른 구성 요소들을 알아서는 안 됨.
3. 변경이 일어날 때, 변경 통지에 대한 처리 방법이 구현되어 있어야 함.
(모델과 같이 변경이 일어날 때, 누군가에게 변경을 알려주는 방법을 구현해야 함.)

### Controler
데이터와 사용자 인터페이스(UI)를 잇는 다리 역할을 함.
규칙:
모델이나 뷰에 대해서 알고 있어야 함.
2. 모델이나 뷰의 변경을 모니터링 해야 함.
(모델이나 뷰의 변경 통지를 받으면 이를 해석, 각각의 구성 요소를 통지해야 하고, 앱의 메인 로직은 컨트롤러가 담당하게 됨.)

## 왜 MVC를 써야 할까?
사용자가 보는 페이지(View), 데이터 처리(Model), 이 두 가지를 중간에서 제어(Controler)
이 세 가지로 구성되는 하나의 앱을 만들면 각각 맡은 역할에만 집중을 할 수 있게 됨.
= 서로 분리되어서 각자의 역할에 집중, 개발하게 되면 생기는 장점(유지보수성/확장성/유연성 증가, 중복코딩 문제점 사라짐)이 많음.
