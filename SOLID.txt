# SOLID (SRP, OCP, LSP, ISP, DIP)

## SRP
Single Responsibility Principle (단일 책임 원칙)은 객체 지향 프로그래밍에서 클래스는 단 하나의 기능만 가져야 하며, 변경해야 할 이유는 단 하나여야 한다는 설계 원칙
### 장점
1. 변경의 이유 최소화: 하나의 클래스가 여러 책임을 가지면 한 기능의 변경이 다른 기능에 영향을 끼치는 결합이 발생 -> SRP를 적용하면 기능 변경 시 관련된 코드만 수정하면 됨
2. 높은 응집도: 클래스 내부 기능들이 서로 밀집하게 연관 -> 코드 이해와 유지보수 쉬워짐
### 만약 위반한다면?
코드 수정이 어려워짐

## OCP
Open Closed Principle (개방 폐쇄 원칙)은 객체의 확장은 개방적으로, 객체의 수정은 폐쇄적으로 대해야 한다는 설계 원칙
### 장점
1. 객체 간의 의존성 최소화: 코드 변경에 따른 영향력 낮춤
2. 새 클래스를 추가하는 상황에서 기존 메서드나 다른 클래스의 코드가 변경되지 않을 수 있다.
### 만약 위반한다면?
기존 코드를 건드리면, 그 코드를 사용하는 모든 곳을 다시 테스트해야 함.

## LSP
Liskov Substitution Principle (리스코프 치환 원칙)은 자식 클래스는 언제나 부모 클래스를 대체할 수 있어야 한다는 설계 원칙
### 장점
1. 다형성을 통한 확장성: 부모 클래스 타입으로 코드를 짜 두었을 때, 어떤 자식 클래스가 들어와도 코드를 수정할 필요가 없음
2. 객체에 대한 신뢰도 향상: 부모 타입을 상속받은 객체들에 대해서, 어떤 기능이 어떻게 동작하는지에 대한 예측이 가능해져서 협업과 디버깅이 편해짐.
### 만약 위반한다면?
부모 클래스에서 잘 돌아가던 기능이 특정 자식 케이스에서는 에러를 던지거나 이상한 값을 반환할 수 있음.

## ISP
Interface Segregaion Principle (인터페이스 분리 원칙)은 사용하지 않는 메서드에 의존하도록 강제하지 말라는 설계 원칙
### 장점
1. 코드의 가독성/재사용성 향상: 인터페이스가 잘게 잘 쪼개져 있으면 구현해야 할 기능이 명확해지고, 필요한 인터페이스만 모아 조합해서 쓰기 좋음
2. 불필요한 리컴파일링 방지: 인터페이스에 기능 하나를 추가하면 그 인터페이스를 사용하는 모든 클래스를 다시 컴파일해야 하는데, ISP를 지키면 상관 없는 기능이 변했을 때 코드가 영향을 받지 않음
### 만약 위반한다면?
어떤 클래스가 인터페이스를 상속받았는데, 쓰지도 않는 메서드를 억지로 구현하게 되면서 잠재적 에러의 원인이 되기도 함

## DIP
Dependency Inversion Principle (의존 역전 원칙)은 고수준 모듈은 저수준 모듈에 의존해서는 안 되며, 둘 다 추상화에 의존해야 한다는 설계 원칙
### 장점
1. 결합도 감소와 교체 용이성: 실제 구현체가 바뀌더라도 인터페이스를 보고 있는 메인 코드는 수정할 필요가 없음 = 부품 교체가 아주 쉬움
2. 시스템의 확장성: 새로운 기능을 추가할 때 기존의 메인 로직을 건드리지 않고, 인터페이스를 상속받는 새로운 클래스만 만들기만 하면 됨
### 만약 위반한다면?
하위 클래스 하나를 수정했더니 그와 연결된 상위 클래스를 통째로 다시 고치고 테스트하게 됨
